# 多人联机贪吃蛇游戏开发方案

## 1. 项目背景与目标

### 1.1 项目背景

传统贪吃蛇玩法简单、上手快，但单机模式缺少持续对抗性与社交性。多人联机版本可以把“实时对战、抢资源、躲避碰撞、最终结算排名”结合起来，形成更强的竞技体验与观赏性。

### 1.2 项目目标

- 支持多人同时在线进入同一局游戏。
- 通过 WebSocket 实现低延迟实时同步。
- 在固定时长对局结束后进行积分结算，按排名决出胜者。
- 提供稳定的房间系统、匹配系统、观战与结算展示。
- 玩法保持经典贪吃蛇核心，同时加入少量合理拓展，不做过度离奇设计。

### 1.3 核心体验定位

- **轻竞技**：每局 3～8 分钟，节奏快。
- **高对抗**：通过撞头、包围、抢食、抢道具形成博弈。
- **强结算**：每局有明确排名、积分、击杀数、存活时长等结果。

## 2. 整体架构

### 2.1 架构原则

- **前后端分离**：前端负责展示与交互，后端负责房间管理、对局裁决与数据持久化。
- **实时通信优先**：游戏过程使用 WebSocket，避免轮询带来的延迟与成本。
- **服务职责清晰**：登录、匹配、房间、游戏、结算、排行榜模块解耦。

### 2.2 系统架构

#### 前端

- 登录/注册页
- 大厅页
- 房间页
- 游戏页
- 结算页
- 排行榜页
- 个人中心页

#### 后端

- 用户认证服务
- 房间管理服务
- 游戏逻辑服务
- 实时通信网关（WebSocket）
- 结算与积分服务
- 排行榜服务
- 日志与监控服务

#### 数据层

- 关系型数据库：用户、房间、对局记录、结算数据、积分记录
- Redis：在线状态、房间缓存、游戏帧状态、临时匹配队列
- 对象存储（可选）：头像、皮肤资源、战绩截图

### 2.3 WebSocket 通信设计

#### 建议通信内容

- 玩家进入房间、退出房间
- 游戏开始、倒计时、暂停、结束
- 玩家方向输入、加速输入、技能触发
- 蛇身位置同步、食物刷新、道具刷新
- 撞击、淘汰、击杀、积分变化
- 房间内聊天、系统广播

#### 通信机制

- **客户端上报输入**，服务端统一裁决。
- **服务端按固定 Tick 广播状态**，例如 10~20 次/秒。
- **状态校验**：服务端作为唯一权威，防止作弊。
- **断线重连**：短时断线可恢复到当前房间状态。

### 2.4 典型流程

1. 用户登录进入大厅。
2. 创建房间或加入房间。
3. 房主点击开始，进入准备倒计时。
4. 对局开始，服务端按 Tick 推送游戏状态。
5. 玩家撞墙、撞自己、撞其他蛇头时淘汰。
6. 时间结束后自动结算，按积分排名。
7. 展示本局数据并可一键再来一局。

## 3. 详细功能模块拆分

### 3.1 用户模块

#### 功能需求

- 用户注册、登录、退出登录
- 第三方登录（可选，如微信/QQ/Google）
- 头像、昵称、个性签名设置
- 玩家等级、经验、段位、历史战绩展示
- 在线状态展示
- 黑名单/禁言管理（可选）

#### 详细功能点

- 账号密码登录、验证码登录
- 找回密码
- 昵称唯一性校验
- 头像上传与裁剪
- 基础资料修改
- 战绩查询：胜率、最高分、平均排名、击杀数
- 历史对局回放索引（可选）

### 3.2 大厅模块

#### 功能需求

- 展示在线人数、正在进行的房间数
- 快速匹配入口
- 创建房间入口
- 查看热门房间列表
- 查看公告、活动、版本更新说明
- 查看排行榜、个人战绩入口

#### 详细功能点

- 一键快速匹配
- 自定义筛选房间：人数、模式、密码房
- 最近加入房间快捷入口
- 推荐新手房间/高手房间

### 3.3 房间模块

#### 功能需求

- 创建房间、加入房间、退出房间
- 房间密码、房间人数上限、观战开关
- 房主权限管理
- 准备状态管理
- 聊天、表情、快捷消息
- 队伍设置（可选）

#### 详细功能点

- 房间基础信息：房间名、模式、地图、时长、人数上限
- 玩家列表：头像、昵称、准备状态、网络状态、段位
- 房主操作：踢人、锁房、开始游戏、设置地图
- 断线保留席位：短时掉线可重新加入
- 房间内准备倒计时：防止临开局频繁退出

### 3.4 游戏模块

#### 功能需求

- 经典贪吃蛇移动、转向、成长
- 玩家蛇头撞击淘汰
- 时间结束结算排名
- 积分、击杀、存活时间、吞食数量统计
- 地图边界、障碍物、随机食物生成
- 加速机制、道具机制、碰撞判定

#### 详细功能点

- **移动规则**：蛇按固定速度前进，玩家只能改变方向。
- **成长规则**：吃到食物后蛇身增长，体积变大。
- **碰撞规则**：
  - 撞墙淘汰
  - 撞自己淘汰
  - 蛇头撞蛇头，速度优先或长度优先判定胜负
  - 蛇头撞到对方蛇身，可导致撞击方淘汰或双方受伤，建议采用经典“头撞身淘汰”
- **积分规则**：
  - 吃食物得基础分
  - 淘汰他人得额外分
  - 连续存活有生存分
  - 最终按积分排序，若同分则比较击杀数、长度、存活时间
- **时间规则**：
  - 固定局时，如 5 分钟
  - 结束前 30 秒提示警报
  - 结束时停止移动并进入结算
- **地图元素**：
  - 普通食物
  - 高分食物（少量刷新）
  - 加速道具（短时间提升速度）
  - 护盾道具（抵消一次撞击，建议稀有）
  - 随机障碍物（数量少，避免过度复杂）

### 3.5 结算模块

#### 功能需求

- 对局结束后生成积分榜
- 显示第一名获胜者
- 展示个人表现与奖励
- 统计本局击杀、吞食、存活时长、最高连胜
- 支持再次开局、返回大厅

#### 详细功能点

- 排名列表：名次、昵称、积分、击杀、长度、存活时间
- 胜者特效展示
- MVP 评选：综合表现最优者
- 奖励发放：经验、金币、段位分（可选）
- 本局数据归档，支持复盘

### 3.6 排行榜模块

#### 功能需求

- 全服积分排行
- 周榜、月榜、赛季榜
- 击杀榜、胜率榜、最长存活榜
- 好友排行榜

#### 详细功能点

- 多维度切换
- 过滤时间范围
- 展示名次变化趋势

### 3.7 社交与运营模块

#### 功能需求

- 好友系统
- 邀请链接/邀请码
- 战绩分享
- 举报与反馈
- 活动公告与新手引导

#### 详细功能点

- 房间邀请一键复制
- 对局结束分享海报
- 举报恶意挂机、辱骂、作弊
- 新手任务与引导奖励

## 4. 玩法设计建议

### 4.1 基础玩法

- 玩家控制蛇移动吃食物变长。
- 在有限时间内尽可能获取高分。
- 通过撞击敌方蛇头或诱导其撞墙来获得优势。

### 4.2 合理拓展玩法

- **地图模式**：普通地图、障碍地图、狭窄竞技地图。
- **道具机制**：加速、护盾、磁铁吸附食物、短暂无敌。
- **击杀奖励**：淘汰对手后掉落大量食物。
- **连杀奖励**：连续淘汰可获得额外积分。
- **危险区域**：随时间缩小安全区，提升对抗强度。
- **复活机制（可选）**：在限次数模式中允许短暂复活，适合娱乐局。

### 4.3 不建议的设计

- 过强技能系统
- 复杂职业分工
- 过多随机事件影响公平性

## 5. 各页面 UI/UX 设计规范

> UI/UX规范只是一起生成了，并不需要按照，**以前端人员自己的设计为主**
>
> 只有**页面功能需保持**。

### 5.1 登录页

#### 页面功能

- 登录、注册、找回密码
- 第三方登录入口（可选）
- 新手游客体验入口（可选）

#### UI/UX 规范

- 采用居中卡片式布局，突出操作入口
- 主按钮高对比、单一核心动作
- 错误提示明确，不遮挡输入区
- 支持记住密码、自动登录
- 新用户引导文案简洁清晰

### 5.2 房间页

#### 页面功能

- 创建房间、加入房间、快速匹配
- 查看房间列表与房间详情
- 房主管理房间设置
- 准备、聊天、邀请好友

#### UI/UX 规范

- 左侧为房间列表，右侧为房间详情或创建面板
- 房间状态标签清晰：等待中、游戏中、已满、密码房
- 玩家准备状态高亮显示
- 房主权限按钮仅对房主可见
- 准备按钮固定显眼，降低误操作成本

### 5.3 游戏页

#### 页面功能

- 实时对局
- 蛇体与食物渲染
- 小地图、计时器、积分榜、排名提示
- 聊天与系统消息
- 道具栏、技能栏（如有）

#### UI/UX 规范

- 游戏画布居中，占据主视觉区域
- 顶部固定显示倒计时、当前排名、个人分数
- 右侧显示实时积分榜与玩家列表
- 左下角显示聊天框，默认折叠
- 关键事件提示要明显但不干扰操作
- 死亡/淘汰提示采用高辨识度视觉反馈

### 5.4 结算页

#### 页面功能

- 展示本局排名
- 突出第一名获胜者
- 展示个人数据与奖励
- 再来一局、返回大厅、分享结果

#### UI/UX 规范

- 中央展示冠军信息与本局摘要
- 排名列表按名次纵向排列
- 对个人数据使用卡片化呈现
- 结算动画简洁有仪式感，不宜过长
- 提供一键继续游戏按钮，减少流失

## 6. 技术实现建议

> 只是建议，可不用管，文档撰写可参考

### 6.1 前端技术建议

- React / Vue + TypeScript
- Canvas / WebGL 渲染游戏画面
- WebSocket 客户端
- 状态管理：Pinia / Redux / Zustand
- UI 组件库：Ant Design / Element Plus / Tailwind 体系

### 6.2 后端技术建议

- Node.js / Java / Go 均可
- WebSocket 网关独立部署
- Redis 做房间状态与实时缓存
- MySQL / PostgreSQL 存储用户与战绩
- 定时任务处理结算、排行榜刷新

### 6.3 关键非功能要求

- 低延迟、高并发、可横向扩展
- 断线重连与状态恢复
- 防作弊与服务端权威判定
- 可观测性：日志、监控、告警

---

## 7. 总结

本项目应以“轻量、实时、竞技、易上手”为核心。整体设计保持经典贪吃蛇的简单规则，同时通过多人实时对抗、限时结算、积分排名、少量道具与地图变化增强可玩性。这样既能保证开发可控，也能带来较强的用户留存与社交传播能力。
